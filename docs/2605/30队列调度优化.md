# 消息队列调度优化：减少联系人切换 + 同联系人跳过重复搜索

**日期**：2026-05-30
**修改文件**：`scripts/message_queue.py`

## 1. 背景与问题

后台 HTTP API（`scripts/app.py`）通过 `MessageQueue` 串行发送消息。常见用法是「同时给多个群发一文一图」，Agent 会发两次请求：

- 第 1 次：给 `a`、`b` 群发文字
- 第 2 次：给 `a`、`b` 群发图片

旧实现使用 `queue.Queue()`（纯 FIFO），worker 严格按入队顺序处理，于是队列为：

```
a文字 → b文字 → a图片 → b图片
```

发送时会话切换路径为 `a → b → a → b`，**切换 3 次**。每次切换都要重新搜索联系人、进入会话，非常耗时。

## 2. 优化目标

把发往同一联系人的消息**连续发完**再切换到下一个联系人，把切换次数降到最少；同一联系人内部仍保持入队顺序（先文字后图片）。要求**不引入固定延迟**。

## 3. 调度算法：贪心「粘连当前联系人」

把内部存储从 `queue.Queue` 换成 `list + threading.Condition`（线程安全且可重排）。worker 每轮从待发送列表中挑选「下一条」时遵循：

1. 若列表中存在发往**上一个联系人**（`last_contact`）的消息，取其中**最早入队**的那条。
2. 否则取**队头**（整体最早入队的消息）。
3. 发送后把 `last_contact` 更新为本条联系人。

由于每轮都重新挑选，**发送期间新入队的消息会被动态纳入重排**，无需任何等待。

### 效果

上面的入队顺序会被发送为：

```
a文字 → a图片 → b文字 → b图片
```

会话切换从 3 次降为 **1 次**（`a → b`）。

### 为什么不需要延迟

单条消息发送（搜索联系人 + 粘贴 + 回车，或下载图片 + 粘贴）通常耗时 **1~3 秒**，而 Agent 连续两次 HTTP 请求入队的间隔是**毫秒级**。当 worker 还在发 `a文字` 时，`a图片`、`b图片` 早已进入队列；发完 `a文字` 挑下一条时即可粘住 `a`。因此零延迟即可正确聚合。

> 仅当「两次请求间隔 > 第一批全部发完所需时间」（例如先发文字、隔很久才发图片）时无法聚合——但此时后续信息尚未到达，任何方案都只能靠等待，故未引入固定延迟。

## 4. 增强：同联系人连发跳过重复搜索

`WeChatController` 中「搜索联系人」与「发送」是分离的两步：

- `search_contact_result(contact)`：搜索并切换到目标会话
- `send_message_result / send_picture_result / send_file_result(content)`：在**当前会话**发送

因此当连续发往同一联系人时，只需第一条执行搜索，后续同联系人的消息可**跳过搜索**直接发送，进一步节省时间。

实现上用 `session_contact` 记录「当前会话已确认停留的联系人」：

- 仅当 `session_contact == 本条联系人` 时才跳过搜索。
- 任意一步（搜索或发送）失败、或处理中抛异常时，把 `session_contact` 置为 `None`，强制下一条**重新搜索**，避免会话状态不确定导致发错对象。

## 5. 实现要点

| 关注点 | 说明 |
| --- | --- |
| 数据结构 | `self._pending`（list，按入队顺序）+ `self._cond`（`threading.Condition`，兼作锁与条件变量） |
| 线程安全 | `_pending`、`_processing` 的读写均在 `with self._cond` 内进行 |
| 阻塞等待 | 队列为空时 `self._cond.wait(timeout=1)`，1 秒超时用于及时响应 `stop()` |
| 退出 | `stop()` 中置 `running=False` 并 `notify_all()`，worker 尽快退出；`join(timeout=5)` |
| 调度状态 | `last_contact`（排序粘连用）、`session_contact`（跳过搜索用，仅确认停留时有效） |
| 发送间隔 | 每条之间仍 `time.sleep(message_interval)`，降低发送过快被风控的风险 |

## 6. 行为变化与兼容性

- 对外接口未变：`start()` / `add_message(to_list, content, action)` / `get_queue_size()` / `stop()` 签名与语义保持兼容，`app.py` 无需改动。
- `get_queue_size()` 返回的是**待处理**数量（不含正在发送的那一条），与旧 `qsize()` 语义一致。
- `wait_until_empty(timeout)` 重写为等待「待发送列表为空且无正在处理的消息」，修正了旧实现中 `timeout` 分支的逻辑问题。
- 发送的**总条数不变**，只改变发送**顺序**与**搜索次数**。

## 7. 边界与注意事项

1. **会话被人为打断**：若后台发送期间有人手动点开了别的微信会话，「跳过搜索」可能在错误会话发送。当前策略只在「上一条成功且联系人相同」时才跳过，已尽量降低风险；后台自动发送场景下一般不会被打断。
2. **饥饿问题**：理论上若某联系人被持续不断地注入新消息，其他联系人会一直等待。实际批量发送场景为有限条数，不会触发；如需可后续增加「同联系人最多连发 N 条」的防饥饿策略。
3. **未引入聚合延迟窗口**：当前为纯零延迟方案；若未来需要覆盖「请求间隔较大」的聚合，可增加一个默认关闭的 `batch_window` 配置。

## 8. 测试方法

`message_queue.py` 自带 `__main__` 测试：交错入队两个联系人的文字与图片，观察日志中是否被重排为「同一联系人连续发送」，且第二条起出现「复用当前会话，跳过联系人搜索」。

```powershell
python scripts/message_queue.py
```

> 需要微信 PC 客户端已登录，且联系人名称（如「线报转发」「文件传输助手」）真实存在。
